Детальное решение
Причина проблемы (коротко)
Три отдельных бага, у которых общий корень — сравнение по количеству элементов вместо сравнения по идентификатору/времени:
js// Было (примерно так, судя по описанию):
if (messages.length <= prevMessagesCount) {
  return; // "новых сообщений нет"
}
Если сервер всегда отдаёт top-50 (или last-50) — length почти всегда одинаковый → условие ложно-положительное, обновление не происходит.
Разберу оба варианта решения: быстрый патч (polling) и правильное решение (WebSocket). Дам архитектуру и псевдокод под твой стек (FastAPI + vanilla JS), чтобы можно было сразу перенести в реальные файлы, как только пришлёшь их.

Вариант A — Быстрый патч (оставляем polling, чиним логику)
A.1. Чат (внутри треда)
Идея: хранить не количество сообщений, а ID последнего полученного сообщения. Сравнивать по нему. Дополнительно — запрашивать с сервера не «все 50», а только сообщения после последнего известного ID (?after_id=...), это и экономит трафик, и снимает саму проблему с фиксированным лимитом.
Бэкенд — новый/изменённый эндпоинт:
GET /chats/{chat_id}/messages?after_id={last_id}&limit=50
Логика: WHERE chat_id = :chat_id AND id > :after_id ORDER BY id ASC LIMIT 50.
Если after_id не передан — обычная пагинация (последние 50, как сейчас).
Фронтенд — polling внутри чата:
jslet lastMessageId = 0; // или id последнего отрисованного сообщения

async function pollMessages(chatId) {
  const res = await fetch(`/chats/${chatId}/messages?after_id=${lastMessageId}`);
  const newMessages = await res.json();

  if (newMessages.length > 0) {
    newMessages.forEach(renderMessage);          // дорисовываем только новые
    lastMessageId = newMessages[newMessages.length - 1].id;
    scrollToBottom();
  }
}

setInterval(() => pollMessages(currentChatId), 3000);
Ключевое отличие от текущей логики: мы не сравниваем длину массива, мы добавляем дельту. Даже если в чате уже 500 сообщений — сервер вернёт только те, что появились после lastMessageId, и они всегда отрисуются.
Важно: lastMessageId нужно инициализировать при первой загрузке чата значением ID последнего сообщения из истории, а не нулём — иначе при входе в чат подтянутся заново все старые сообщения при первом polling-тике.
A.2. Список чатов (главная страница чатов)
Та же проблема — нужен маркер для сравнения, но здесь маркер не ID сообщения, а timestamp последнего обновления чата (updated_at на чате, который обновляется при каждом новом сообщении в нём).
Бэкенд: у chats уже должно быть поле updated_at (обновляется триггером или в коде при вставке сообщения). Эндпоинт:
GET /chats?since={last_check_timestamp}
Возвращает чаты, где updated_at > since, либо просто весь список — можно не оптимизировать это отдельно, если чатов немного, важнее сам факт diff-рендеринга.
Фронтенд:
jslet chatSnapshots = {}; // chat_id -> updated_at

async function pollChatList() {
  const res = await fetch('/chats');
  const chats = await res.json();

  chats.forEach(chat => {
    const prevUpdatedAt = chatSnapshots[chat.id];
    if (!prevUpdatedAt || chat.updated_at !== prevUpdatedAt) {
      updateChatListItem(chat); // перерисовываем именно эту строку списка
      chatSnapshots[chat.id] = chat.updated_at;
    }
  });
}

setInterval(pollChatList, 3000);
Не перерисовывать список целиком (innerHTML = '' + заново всё) — это сбивает скролл и моргает UI. Обновлять точечно только изменившиеся элементы (по chat.id, найти DOM-узел через data-chat-id, обновить текст последнего сообщения и время).
A.3. Бейдж непрочитанных в навбаре
Проблема была в том, что этот polling жил только внутри страницы чатов. Решение — вынести на уровень всего приложения.
Бэкенд: отдельный лёгкий эндпоинт:
GET /chats/unread-count
→ { "count": 4 }
(считает COUNT(*) сообщений где sender_id != current_user_id AND is_read = false, сгруппированных по чатам пользователя).
Фронтенд: polling инициализируется один раз в общем layout-скрипте (том, что грузится на каждой странице — видимо main.js/app.js, а не внутри chat.js, который живёт только пока открыта страница чатов):
js// глобальный скрипт, подключаемый везде
async function pollUnreadBadge() {
  const res = await fetch('/chats/unread-count');
  const { count } = await res.json();
  updateNavbarBadge(count); // показать/скрыть/обновить цифру
}

setInterval(pollUnreadBadge, 5000);
pollUnreadBadge(); // сразу при загрузке страницы
Плюс: когда пользователь открывает чат и дочитывает его — нужно дергать POST /chats/{chat_id}/mark-read, чтобы бейдж корректно уменьшался, а не просто расти.
Итог по варианту A
Три независимых, но однотипных фикса. Не требует трогать бэкенд-инфраструктуру (WebSocket), укладывается в текущий стек. Задержка в 3–5 секунд остаётся (это не «настоящий realtime», а «почти realtime»).

Вариант B — Правильное решение (WebSocket)
Если хочешь, чтобы это реально ощущалось как Telegram/Instagram (сообщение появляется мгновенно, а не через 3 секунды) — нужен WebSocket. Это больше работы, но кардинально решает проблему в корне: сервер сам пушит данные клиенту, никакого polling и сравнения количества вообще не нужно.
B.1. Бэкенд (FastAPI)
Connection Manager — центральный реестр активных соединений:
python# app/ws_manager.py
from fastapi import WebSocket
from typing import Dict, Set

class ConnectionManager:
    def __init__(self):
        # user_id -> набор активных сокетов (юзер может быть открыт в неск. вкладках)
        self.active_connections: Dict[int, Set[WebSocket]] = {}

    async def connect(self, user_id: int, websocket: WebSocket):
        await websocket.accept()
        self.active_connections.setdefault(user_id, set()).add(websocket)

    def disconnect(self, user_id: int, websocket: WebSocket):
        self.active_connections.get(user_id, set()).discard(websocket)

    async def send_to_user(self, user_id: int, message: dict):
        for ws in self.active_connections.get(user_id, set()):
            await ws.send_json(message)

manager = ConnectionManager()
WS-эндпоинт:
python# app/routers/ws.py
@router.websocket("/ws/{user_id}")
async def websocket_endpoint(websocket: WebSocket, user_id: int, token: str = Query(...)):
    # тут проверка token (тот же JWT/сессия, что и в обычном auth) — 
    # ОБЯЗАТЕЛЬНО, иначе кто угодно сможет слушать чужие сообщения
    verify_ws_token(token, user_id)
    await manager.connect(user_id, websocket)
    try:
        while True:
            await websocket.receive_text()  # держим соединение живым (ping/pong)
    except WebSocketDisconnect:
        manager.disconnect(user_id, websocket)
При отправке сообщения (в существующем POST /chats/{id}/messages) — добавляем push:
python@router.post("/chats/{chat_id}/messages")

Один сокет на пользователя (не на чат) — так проще: через него прилетают и новые сообщения (с любым chat_id), и обновления бейджа, и обновления списка чатов. Не нужно плодить по сокету на каждый тред.
B.2. Фронтенд
Один глобальный WebSocket на всё приложение, открывается сразу после логина (в общем layout-скрипте):
js// app.js — грузится на каждой странице
let socket;

function connectSocket() {
  socket = new WebSocket(`ws://localhost:8000/ws/${currentUser.id}?token=${authToken}`);

  socket.onmessage = (event) => {
    const data = JSON.parse(event.data);

    if (data.type === "new_message") {
      // если пользователь сейчас в этом чате — дорисовать сообщение
      if (currentOpenChatId === data.chat_id) {
        renderMessage(data.message);
        scrollToBottom();
      }
      // в любом случае — обновить список чатов (последнее сообщение/время)
      updateChatListItem(data.chat_id, data.message);
    }

    if (data.type === "unread_count") {
      updateNavbarBadge(data.count);
    }
  };

  socket.onclose = () => {
    // переподключение при обрыве связи
    setTimeout(connectSocket, 2000);
  };
}

connectSocket();
Никакого setInterval, никакого сравнения количества — сервер сам говорит «вот новое сообщение» в момент его появления.
B.3. Что с фото/голосовыми/геолокацией (из ТЗ)
Тот же механизм: тип сообщения (photo/voice/location) просто передаётся полем type в том же new_message пуше — рендер на фронте просто выбирает нужный шаблон отображения по data.message.type.
