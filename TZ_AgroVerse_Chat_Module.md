# Техническое задание
## Модуль «Чаты» и изменение логики выбора драйвера — AgroVerse

**Версия:** 1.0
**Проект:** AgroVerse (FastAPI + PostgreSQL back, Vanilla JS front)
**Автор ТЗ:** составлено по описанию заказчика

---

## 1. Цель

Добавить в платформу систему чатов между тремя ролями — **покупатель ↔ фермер**, **покупатель ↔ драйвер**, **драйвер ↔ фермер** — привязанных к конкретному заказу, с модерацией со стороны админа. Параллельно изменить логику выбора драйвера: сейчас выбор драйвера в заказе = мгновенная автоматическая отправка заказа драйверу. Это отвязывается: выбор драйвера становится «мягким» действием (просто выбор кандидата), а фактическая передача заказа драйверу происходит отдельным явным действием («Заказать этого драйвера») после того, как стороны договорились в чате.

---

## 2. Глоссарий

| Термин | Значение |
|---|---|
| Заказ | Order — покупка одного или нескольких товаров у фермера |
| Способ доставки | «Самовывоз» / «Доставка фермером» (если фермер это включил) / «Доставка драйвером» |
| Чат | Отдельный тред переписки, привязанный к заказу и паре участников |
| Тип чата | `buyer_farmer`, `buyer_driver`, `driver_farmer` |
| Кандидат-драйвер | Драйвер, выбранный покупателем для обсуждения, но ещё не назначенный на заказ |
| Назначение драйвера | Момент, когда заказ реально передаётся драйверу в работу |

---

## 3. Как работает сейчас (as-is)

1. В карточке товара на рынке покупатель выбирает способ получения: «Самовывоз» / «Доставка фермером» / «Доставка драйвером».
2. При выборе «Доставка драйвером» → драйвер выбирается **и в этот же момент** заказ автоматически уходит этому драйверу в работу.
3. У покупателя есть раздел «Мои заказы».
4. У драйвера нет раздела «Чаты» / переписки вообще.
5. Прямого канала связи между покупателем, фермером и драйвером нет — только через админа или телефон (что и нужно исключить).

---

## 4. Как должно работать (to-be)

### 4.1. Новый раздел навбара — «Чаты»

- Добавляется новый пункт в навбаре: **«Чаты»** (доступен покупателю, фермеру, драйверу; админу — отдельный интерфейс модерации, см. п. 7).
- Раздел показывает список чатов пользователя, сгруппированных по заказам, с бейджем непрочитанных сообщений.
- Открытие конкретного чата из уведомления/кнопки — сразу открывает нужный тред внутри этого раздела (deep link).

### 4.2. Сценарий «Покупатель ↔ Фермер»

1. Покупатель создал заказ, ещё не оплатил. В «Мои заказы» у заказа появляется кнопка **«Чат с фермером»**.
2. При первом нажатии на кнопку — модальное окно с предупреждением:
   > «Этот чат сохраняется и доступен администраторам платформы для просмотра диалога.»
   С кнопками «Понятно, продолжить» / «Отмена».
3. После подтверждения:
   - Создаётся новый чат типа `buyer_farmer`, привязанный к заказу.
   - Покупателя переносит в раздел «Чаты», сразу в этот тред.
4. У фермера в его разделе «Чаты» появляется новый чат с этим покупателем (с привязкой к заказу — название/фото товара, номер заказа).
5. Общение свободное, с учётом правил из п. 6.

### 4.3. Сценарий «Покупатель ↔ Драйвер» + изменение логики выбора

1. На карточке товара при выборе доставки драйвером — **изменение логики**: выбор драйвера теперь **не** отправляет заказ драйверу автоматически. Это просто фиксирует «кандидата» на этот заказ.
2. После выбора кандидата у заказа в «Мои заказы» появляется кнопка **«Чат с драйвером»** (по аналогии с фермером, под кнопкой чата с фермером).
3. Нажатие → то же предупреждение про модерацию → создаётся чат типа `buyer_driver` → покупателя переносит в раздел «Чаты» в новый (отдельный от чата с фермером) тред.
4. У драйвера (у которого до этого не было раздела «Чаты» вообще — теперь появляется) — новый чат с этим покупателем.
5. **Шапка чата на стороне драйвера** отображает карточку с информацией о заказе:
   - Что покупается (товар, кол-во)
   - Откуда забрать (адрес/точка фермера)
   - Куда доставить (адрес покупателя)
   - От кого заказ (покупатель)
6. Внутри этого чата (сторона драйвера) — кнопка **«Начать чат с фермером»** (нужна драйверу, чтобы уточнить детали забора товара). Ведёт в отдельный чат `driver_farmer`, не путается с чатом покупателя.
7. Внутри этого же чата (сторона покупателя) — кнопка **«Заказать этого драйвера»**. Это и есть точка фактического назначения драйвера на заказ (замена автоматической логики из п.1).
8. Внутри этого же чата (сторона драйвера) — симметричная кнопка **«Начать чат с покупателем»** (на случай если чат ещё не начат/закрыт).

### 4.4. Сценарий «Драйвер ↔ Фермер»

- Инициируется только драйвером (кнопка описана в 4.3.6).
- Отдельный чат `driver_farmer`, привязанный к тому же заказу.
- Назначение чата — согласовать место/время забора товара.
- Правила и модерация те же (п. 6).

### 4.5. После оплаты

1. Покупатель и драйвер договорились в чате → покупатель возвращается в «Мои заказы» → оплачивает заказ.
2. После оплаты покупатель либо:
   - идёт в раздел «Чаты» → в чате с драйвером нажимает **«Заказать этого драйвера»** (если ещё не нажимал), либо
   - в «Мои заказы» есть кнопка **«Выбрать другого драйвера»** — **эта логика не меняется**, остаётся как есть сейчас.
3. Реальное назначение драйвера на доставку происходит только через «Заказать этого драйвера» (из чата) — до этого момента заказ «висит» без исполнителя-драйвера.

## 5. Правила чата (единые для всех типов чатов)

**Разрешено отправлять:**
- Текст
- Фото
- Голосовые сообщения
- Геолокацию (точку на карте)

**Запрещено:**
- Передача номера телефона в любом виде (цифрами, словами, с разделителями и т.д.)

**Реализация запрета:**
- Серверная проверка текста сообщения на паттерны, похожие на номер телефона (регулярные выражения + нормализация: пробелы, дефисы, "плюс", слова-цифры и т.п.).
- При обнаружении — сообщение **не отправляется**, пользователю показывается предупреждение: «Обмен номерами телефона запрещён правилами чата».
- Правила чата отображаются баннером сверху чата и/или в модальном окне при первом входе в чат («Согласно правилам платформы запрещён обмен номерами телефона. Весь диалог сохраняется и доступен администраторам.»).

## 6. Модерация (Админ)

- В админ-панели — раздел просмотра всех чатов платформы (фильтр по заказу / по пользователю / по типу чата).
- Админ **только просматривает** переписку (read-only), без права писать в чат от своего имени (если нужно иное — уточнить отдельно).
- Хранится полная история сообщений всех типов (текст/фото/голос/локация) без возможности удаления пользователем задним числом (для целей модерации).

## 8. API — новые эндпоинты (ориентировочно)

```
POST   /chats                          — создать чат (order_id, type)
GET    /chats                          — список чатов текущего пользователя
GET    /chats/{chat_id}/messages       — история сообщений (пагинация)
POST   /chats/{chat_id}/messages       — отправить сообщение (text/photo/voice/location)
POST   /orders/{order_id}/select-driver-candidate   — выбор кандидата (без назначения)
POST   /orders/{order_id}/assign-driver             — фактическое назначение («Заказать этого драйвера»)

# Админка
GET    /admin/chats                    — список всех чатов (фильтры)
GET    /admin/chats/{chat_id}/messages — просмотр переписки
```

Реалтайм: рекомендуется WebSocket (например `/ws/chats/{chat_id}`) для доставки сообщений в реальном времени; при отсутствии WS-инфраструктуры — переход на polling с интервалом 3–5 сек как временное решение.

## 9. UI/UX требования

- Новый пункт навбара «Чаты» — с индикатором непрочитанных (счётчик).
- Список чатов: сортировка по последнему сообщению, иконка роли собеседника (фермер/драйвер/покупатель), название/номер заказа.
- В самом чате — шапка с карточкой заказа (для чата покупатель↔драйвер и драйвер↔фермер обязательно: откуда/куда/что/кто).
- Кнопки действия («Заказать этого драйвера», «Начать чат с фермером» и т.д.) — контекстные, показываются только нужной роли в нужном чате (см. таблицу ниже).

|---|---|---|

## 10. Критерии приёмки

1. Выбор драйвера на карточке товара **не** отправляет заказ драйверу автоматически.
2. Заказ уходит драйверу в работу только после нажатия «Заказать этого драйвера» в чате.
3. В навбаре у покупателя, фермера и драйвера появляется раздел «Чаты».
4. У драйвера впервые появляется собственный интерфейс чатов (ранее отсутствовал).
5. Кнопки «Чат с фермером» / «Чат с драйвером» в «Мои заказы» создают отдельные, не пересекающиеся между собой треды.
6. При первом открытии чата показывается предупреждение о модерации админом.
7. В чате работает отправка текста, фото, голосовых, геолокации.
8. Попытка отправить номер телефона (в любом распознаваемом формате) блокируется на сервере с понятным сообщением пользователю.
9. Админ может открыть любой чат в режиме просмотра.
10. Существующая логика «Выбрать другого драйвера» после оплаты не изменена.

## 11. Допущения и открытые вопросы (требуют подтверждения заказчика)

1. **Название раздела в навбаре** — предложено «Чаты» (можно заменить на «Сообщения»/«Диалоги» по вкусу — на функциональность не влияет).
2. **Права админа в чате** — заложено как «только просмотр». Если нужно, чтобы админ мог писать в чат (например, для вмешательства при споре) — это отдельная доработка.
3. **Хранение фото/голосовых** — предполагается объектное хранилище (S3-совместимое или локальная папка + раздача через FastAPI static). Нужно уточнить, что уже есть в инфраструктуре.
4. **Формат геолокации** — предполагается точка (широта/долгота) с отображением на встроенной карте в чате.
5. **Блокировка номеров телефона** — регулярное выражение может давать ложные срабатывания (например, на артикул товара из 10+ цифр). Нужно протестировать на реальных данных.
6. **Реалтайм** — WebSocket vs polling: зависит от того, что проще развернуть в текущем docker-compose стеке. Если WS не критичен на первой итерации — можно стартовать с polling.
